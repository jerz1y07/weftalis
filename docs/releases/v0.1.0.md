# Weftalis v0.1.0 Release Notes (Draft)

> Release status: candidate awaiting maintainer approval. No `v0.1.0` tag or GitHub Release has been created.

Weftalis v0.1.0 is the first public release candidate for an open registry of reusable and verifiable workflows. It establishes a transparent, file-based path from a Workflow Package to static Registry data and a read-only public website. It does not execute Workflows.

## Highlights

### Workflow Package Specification v0.1

The Weftalis Workflow Package Specification v0.1 defines a small metadata wrapper for n8n and Dify exports. A Package declares its identity, version, runtime, inputs, outputs, dependencies, permissions, Human Review checkpoints, safety metadata, testing status, license, documentation, source file, and safe examples.

The specification is a description format, not a workflow language or execution engine. The original platform export remains authoritative for execution.

### Static Validator

The local Validator parses Package files as data and checks the manifest against JSON Schema Draft 2020-12. It also checks SPDX expressions, referenced paths and files, several common secret patterns, basic n8n and Dify export structure, and a limited set of statically detectable capabilities against declared permissions and safety metadata.

The Validator never runs nodes, evaluates embedded code, contacts workflow platforms, or resolves remote resources. Unknown nodes produce warnings for manual review.

### Registry Builder and generated Registry

Registry Builder discovers direct child Packages under `packages/`, invokes the Validator, normalizes public metadata for accepted Packages, and writes `registry/registry.json` and `registry/rejected.json`. Registry verification regenerates data in a temporary directory and compares all content except documented timestamp fields.

The generated Registry contains public metadata, not complete Workflow exports. Website synchronization copies the accepted Registry into a committed generated file and checks its structure and identity consistency.

### Public static website

The public GitHub Pages site at <https://jerz1y07.github.io/weftalis/> provides six static HTML routes: the homepage, Workflow index, collection, contribution guide, and two Workflow detail pages. It also publishes `robots.txt` and `sitemap.xml` with production canonical URLs.

The site is a read-only catalog. Disabled download and source buttons are intentional; there is no upload form, account system, database, API route, or online execution service.

### Initial Workflow Packages

- **Multi-Source Research Assistant** is an n8n example that retrieves user-selected public sources, requires evidence review, and assembles a structured research summary.
- **Human-Reviewed Writing Pipeline** is a Dify example that drafts from supplied material, checks selected claims against a public source, and requires approval before returning final copy.

Both examples declare outbound network data transfer and Human Review checkpoints. The Dify example also declares code execution. They contain no credentials and are not automatically connected to each other.

### Contributions and CI

Contributors can propose Packages and focused code changes through GitHub Pull Requests using the template and review guides. CI installs locked dependencies with lifecycle scripts disabled, then checks Validator types, tests, and build; Registry Builder types, tests, build, and committed Registry consistency; and website Registry synchronization, lint, and static build.

GitHub Pages uses a separate least-privilege deployment workflow and a `/weftalis` base-path build.

## Security boundary and human review

Validation confirms only that files passed the current automated rules and limited static evidence checks. It does not prove that a Workflow is safe, truthful, useful, legally reusable, compatible, or executable. Secret detection and platform capability detection can both produce false positives and false negatives.

Before reuse, a person must inspect the Package README, manifest, original export, examples, permissions, data flows, external services, license claims, embedded code or prompts, and every declared Human Review checkpoint. Never execute an untrusted Workflow merely because it is listed or marked `valid`.

## Current limitations

- Only n8n and Dify are accepted in specification version 0.1.
- Node recognition is intentionally limited; community, future, indirect, dynamic, or obfuscated behavior may be missed.
- Static checks cannot verify runtime compatibility, actual execution, metadata truthfulness, source quality, service behavior, license ownership, or complete secret absence.
- Registry discovery is local and non-recursive, with one Package version per top-level Package folder.
- Registry timestamps vary between builds and are the only fields ignored by the consistency verifier.
- The website has no search implementation, downloads, source links, API, uploads, authentication, database, analytics, custom domain, or Workflow execution.
- The initial Packages are illustrative examples and not endorsements or production guarantees.

## Local use

Use Node.js `24.18.0` from `.nvmrc`. Complete commands for the Validator, Registry Builder, Registry synchronization, normal website build, and GitHub Pages base-path build are in the repository [README](../../README.md#installation-and-local-use).
